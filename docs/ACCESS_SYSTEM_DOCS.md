# 🩺 HealthX Delegated Access System

**Author:** Ashutosh Kumar Singh

---

# System Overview

The Delegated Access System allows HealthX users to share their health data seamlessly via QR codes.

Users can generate public **"hashes"** (secure tokens) that contain specific permission scopes. When scanned by another user, a bidirectional **Friendship** is established, granting granular, easily revocable access to the initiator's data.

The system utilizes an interception middleware pattern, allowing shared access to existing routes without rewriting standard application logic.

---

# 1. Database Models

| Model | Purpose | Key Fields |
| :--- | :--- | :--- |
| `ShareableHash` | Stores the active public links generated by users for their QR codes. | `hashId` (String, Indexed), `userId` (ObjectId), `actions` (Array), `status` (String) |
| `Friendship` | Maps the active data-sharing relationship and granular toggles. | `ownerId` (ObjectId), `viewerId` (ObjectId), `permissions` (Array of Objects) |
| `Blocklist` | Prevents banned users from re-scanning hashes to regain access. | `userId` (ObjectId), `blockedUserId` (ObjectId) |

# 2. API Reference & Payloads

All routes below require a valid JWT in the following header:

```http
Authorization: Bearer <token>
```

---

## A. Hash Management (Creating the QR)

### 1. Create a Shareable Hash

**Endpoint**

```http
POST /api/access/hash
```

**Description**

Generates a secure random 64-character hex string representing the QR code.

**Request Body**

```json
{
  "actions": ["ALL"]
}
```

**Success Response (201)**

```json
{
  "success": true,
  "message": "QR Hash generated successfully.",
  "data": {
    "hashId": "a1b2c3d4...",
    "userId": "60d5ec...",
    "actions": ["ALL"],
    "status": "ACTIVE"
  }
}
```

---

### 2. Get My Hashes

**Endpoint**

```http
GET /api/access/hash
```

**Description**

Retrieves all hashes generated by the logged-in user.

**Success Response (200)**

```json
{
  "success": true,
  "data": [
    {
      "hashId": "a1b2c3d4...",
      "actions": [
        "SET_GOALS",
        "SEE_NUTRITION"
      ],
      "status": "ACTIVE"
    }
  ]
}
```

---

### 3. Update Hash Status

**Endpoint**

```http
PATCH /api/access/hash/:hashId/status
```

**Request Body**

```json
{
  "status": "UNACTIVE"
}
```

---

### 4. Delete Hash

**Endpoint**

```http
DELETE /api/access/hash/:hashId
```

**Success Response (200)**

```json
{
  "success": true,
  "message": "Hash deleted successfully."
}
```

---

## B. Blocklist Management

### 1. Get Blocklisted Users

**Endpoint**

```http
GET /api/access/blocklist
```

**Description**

Returns basic profile data so the UI can display blocked users cleanly.

**Success Response (200)**

```json
{
  "success": true,
  "data": [
    {
      "userId": "60d5ec...",
      "name": "Rajat",
      "profileImageUri": "/public/uploads/rajat.jpg"
    }
  ]
}
```

---

### 2. Unblock a User

**Endpoint**

```http
DELETE /api/access/blocklist/:blockedUserId
```

**Success Response (200)**

```json
{
  "success": true,
  "message": "User successfully removed from blocklist."
}
```

---

## C. Friendship & Connection Lifecycle

### 1. Scan & Connect (User B scans User A's Hash)

**Endpoint**

```http
POST /api/access/connect/:hashId
```

**Description**

Validates the hash, checks the blocklist, and establishes the Friendship document.

**Success Response (201)**

```json
{
  "success": true,
  "message": "Friend added successfully. You now have access.",
  "data": {
    "ownerId": "UserA_ID",
    "viewerId": "UserB_ID",
    "permissions": [
      {
        "action": "SET_GOALS",
        "isActive": true
      }
    ]
  }
}
```

---

### 2. Update Granular Permissions (User A limits User B)

**Endpoint**

```http
PATCH /api/access/friends/:friendId/permissions
```

**Description**

Modifies the `isActive` boolean for specific actions.

**Request Body**

```json
{
  "action": "SET_GOALS",
  "isActive": false
}
```

---

### 3. Remove a Friend (User A removes User B)

**Endpoint**

```http
DELETE /api/access/friends/:friendId
```

**Description**

Deletes the Friendship document. Does **NOT** block the user.

---

### 4. Block a Friend (User A bans User B)

**Endpoint**

```http
POST /api/access/friends/:friendId/block
```

**Description**

Deletes the Friendship document **AND** inserts User B's ID into the Blocklist.

# 3. Standardized API Responses (Outputs & Errors)

The HealthX backend uses a consistent JSON structure for every route.

The frontend should always check the `success` boolean first.

---

## A. The Standard Error Model (Frontend Catch-All)

Whenever an API call fails (**4xx** or **5xx** status codes), the backend will return this exact format.

The frontend should read the `message` to display a toast or alert to the user.

### Error Response

```json
{
  "success": false,
  "message": "Human-readable error description here."
}
```

---

## B. Specific Error Scenarios to Handle

The frontend must be prepared to handle these specific HTTP status codes and messages.

### 400 Bad Request (Validation Errors)

**Scenario**

User B tries to scan a hash but sends invalid data.

**Response**

```json
{
  "success": false,
  "message": "Please provide an array of actions."
}
```

---

### 401 Unauthorized (Auth / Token Errors)

**Scenario**

The JWT token expired or wasn't sent.

The app should force a logout/re-login.

**Response**

```json
{
  "success": false,
  "message": "Session expired. Please log in again."
}
```

---

### 403 Forbidden (Account Issues)

**Scenario**

User tries to do something but their account is suspended.

**Response**

```json
{
  "success": false,
  "message": "Access denied. Account is suspended."
}
```

---

### 404 Not Found (Missing Data)

**Scenario**

User A tries to delete a hash that doesn't exist or they don't own.

**Response**

```json
{
  "success": false,
  "message": "Hash not found or you do not have permission to modify it."
}
```

---

### 500 Internal Server Error (Backend Crash)

**Scenario**

Database goes down.

**Response**

```json
{
  "success": false,
  "message": "Server error generating hash."
}
```

When `success: true`, the requested data is always nested inside the `data` key.

---

    "userId": "64b1c2d3e4f5a6b7c8d9e0f2",
    "actions": [
      "ALL",
      "SET_GOALS"
    ],
    "status": "ACTIVE",
    "createdAt": "2026-07-14T03:42:20.000Z",
    "updatedAt": "2026-07-14T03:42:20.000Z",
    "__v": 0
  }
}
```

```
